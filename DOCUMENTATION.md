# ADIF Analyzer — Подробная документация

## Обзор проекта

**ADIF Analyzer** — это веб-приложение для анализа ADIF-файлов (Amateur Data Interchange Format), используемых радиолюбителями для хранения логов связи. Приложение формирует статистику по странам, Maidenhead-локаторам и категориям мощности (QRP, QRPp, QRP-X, QRPu).

Результаты ориентированы на таблицы активности [Клуба 72](http://club-72.ru/#qrp).

## Как работает сервер (`app.py`)

### Эндпоинты

| Метод | Путь | Описание |
|-------|------|----------|
| `GET` | `/` | Главная страница (HTML-интерфейс) |
| `POST` | `/analyze` | Загрузка файлов и запуск анализа |
| `GET` | `/status/<job_id>` | Проверка статуса задачи и получение результата |
| `GET` | `/download/<job_id>` | Скачивание результата в виде TXT-файла |

### Жизненный цикл задачи

1. **Загрузка**: Пользователь отправляет форму с позывным, локатором и ADIF-файлами через `POST /analyze`.
2. **Создание задачи**: Сервер генерирует уникальный `job_id` (8 символов hex), создаёт директорию `uploads/<job_id>/` и сохраняет туда файлы.
3. **Асинхронная обработка**: Запускается `threading.Thread`, который вызывает `analyze_files()` из `main_analysis.py`. Статус задачи хранится в словаре `jobs` (потокобезопасно через `threading.Lock`).
4. **Поллинг**: Фронтенд каждые 2 секунды опрашивает `GET /status/<job_id>`. Возможные статусы:
   - `processing` — анализ ещё идёт
   - `done` — результат готов, возвращается текст и `download_id`
   - `error` — произошла ошибка
5. **Скачивание**: По кнопке «Скачать TXT» браузер запрашивает `GET /download/<job_id>`.

### Ограничения

- Максимальный размер загрузки: **100 МБ** (`MAX_CONTENT_LENGTH`).
- Адрес сервера: `0.0.0.0:5000` (по умолчанию) или порт из переменной окружения `PORT`.

## Ядро анализа (`main_analysis.py`)

### Входные данные

- `file_list` — список путей к ADIF-файлам
- `my_loc_str` — Maidenhead-локатор оператора (например, `LO48VI`)
- `output_file` — путь для записи результата
- `power_override` (опционально) — принудительное значение мощности для всех QSO (в ваттах)
- `lang` — язык вывода (`ru` или `en`)

### Алгоритм работы

#### 1. Инициализация

```
Загрузка библиотеки стран (pyhamtools → countryfile)
Создание 5 категорий мощности:
  ALL      — все QSO без фильтра
  QRP      — до 5 Вт включительно
  QRPp     — до 1 Вт
  QRP-X    — до 0.1 Вт (100 мВт)
  QRPu     — до 0.01 Вт (10 мВт)
```

#### 2. Парсинг ADIF-файлов

Для каждого файла:
- Чтение содержимого (UTF-8, с игнорированием ошибок)
- Парсинг через `adif_io.read_from_string()`
- Извлечение из каждой QSO:
  - `call` — позывный собеседника
  - `band` — диапазон (или вычисление из `freq`)
  - `tx_pwr` / `power` / `app_rumlog_power` — мощность передатчика
  - `gridsquare` — Maidenhead-локатор собеседника
  - `country` / `dxcc` — страна

#### 3. Обогащение данных

Если в QSO отсутствует локатор или страна:
- Проверяется кэш (`call_cache`)
- При промахе — запрос к QRZ.com через `pyhamtools.Callinfo.get_lat_long()` и `get_country_name()`
- Результат кэшируется для повторного использования

#### 4. Расчёт расстояния

- Сначала через `pyhamtools.locator.calculate_distance()` (точный расчёт)
- При ошибке — fallback на собственную функцию `calculate_distance_approx()` (формула Хаверсина с приближённой конвертацией локатора в координаты)

#### 5. Категоризация по мощности

Каждое QSO попадает во ВСЕ категории, чей порог ≥ фактической мощности. Например, QSO с мощностью 3 Вт попадёт в ALL, QRP, но не в QRPp.

Для каждой категории подсчитываются:
- Количество QSO
- Множества стран/полей/квадратов (для подсчёта WKD)
- ODX (самая дальняя связь)

#### 6. Диапазоны и ODX

Для каждого диапазона (160M, 80M, 40M, ..., 2M, 70CM) считается:
- Количество QSO
- Самая дальняя связь (ODX)

Если `band` не задан — вычисляется из `freq` по таблице соответствий.

## Фронтенд (`templates/index.html`)

### Возможности

- **Интерфейс**: Тёмная тема (Tailwind-подобная палитра), адаптивный дизайн.
- **Язык**: Переключение RU/EN через кнопки с флагами. Выбор сохраняется в `localStorage`.
- **Загрузка файлов**: Клик по зоне или drag-and-drop. Поддержка нескольких файлов.
- **Мощность**: Опциональное принудительное задание мощности для всех QSO (чекбокс + поле ввода).
- **Асинхронный анализ**: POST-запрос → поллинг статуса каждые 2 секунды → отображение результата.
- **Скачивание**: Кнопка для скачивания результата как TXT.

### JavaScript-логика

1. `submit` → формируется `FormData` → `POST /analyze` → получение `job_id`
2. Цикл опроса `GET /status/<job_id>`:
   - `processing` → ждём 2 сек
   - `done` → показываем результат + ссылку на скачивание
   - `error` → показываем сообщение об ошибке
3. Состояние UI: `spinner` (загрузка), `result` (результат), `error` (ошибка)

## Деплой

### Render.com

Конфигурация в `render.yaml`:
```yaml
services:
  - type: web
    name: adif-analyzer
    runtime: python
    buildCommand: pip install -r requirements.txt
    startCommand: gunicorn app:app --timeout 120
    envVars:
      - key: PYTHON_VERSION
        value: "3.11.0"
```

### Локальный запуск

```bash
pip install -r requirements.txt
python app.py
# Сервер доступен на http://localhost:5000
```

### CLI-режим

Можно запустить анализ напрямую без веб-интерфейса:

```bash
python main_analysis.py <file.adif> [locator]
python main_analysis.py <folder>/ [locator]
```

## Вспомогательные функции

| Функция | Описание |
|---------|----------|
| `parse_power_custom(pwr_str)` | Парсинг строки мощности в float (Ватты) |
| `get_base_call(call)` | Извлечение базового позывного из полного (убирает суффиксы типа `/P`, `/M`) |
| `normalize_locator(loc)` | Нормализация Maidenhead-локатора (4→6 символов, uppercase, без спецсимволов) |
| `get_call_info(call, cic, ...)` | Запрос локатора и страны через pyhamtools с retry-логикой (до 3 попыток) |
| `calculate_distance_approx(loc1, loc2)` | Приближённый расчёт расстояния между локаторами (формула Хаверсина) |
| `fmt_pwr(val)` | Форматирование значения мощности (мВт / Ватт) |
